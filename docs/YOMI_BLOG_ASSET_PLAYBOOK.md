# 요미 블로그 운영 자산 플레이북

작성 기준일: 2026-06-12  
대상: Blogspot 오늘 이슈 자동화, 한국어 생활 의사결정형 블로그, 다른 AI/다른 주제/다른 블로그로 이식할 운영 프롬프트

이 문서는 한 달 동안 다듬은 블로그의 정체성, 글쓰기 기준, 주제 선정법, 품질 게이트, 프롬프트, 발행 안전 규칙을 한 파일로 묶은 운영 자산이다.  
새 AI에게 컨텍스트로 붙여넣거나, 다른 프로젝트에서 "이 정도 기준으로 블로그를 설계하라"는 예시로 쓰는 것을 목표로 한다.

## 1. 이 블로그의 한 문장 정체성

이 블로그는 단순 뉴스 요약 블로그가 아니다.  
정체성은 **오늘 이슈를 독자의 돈, 시간, 불안, 선택 기준으로 번역하는 생활 의사결정 블로그**다.

최종 포지션:

> AI자동화돈 되는 생활 선택 기준 미디어

주요 독자:

- 30~50대 직장인
- 돈, 시간, 생산성, 생활비, 디지털 변화에서 손해 보지 않기 위해 빠른 판단 기준이 필요한 일반 소비자
- 뉴스를 끝까지 읽기보다 "그래서 내게 뭐가 달라지나"를 먼저 알고 싶은 모바일 독자

독자가 글을 읽고 바로 판단해야 하는 것:

- 내가 돈을 더 내야 하는가
- 내가 받을 수 있는 돈이나 환급이 있는가
- 지금 신청, 해지, 환불, 설정 변경을 해야 하는가
- 이 변화가 내 생활, 소비, 업무, 계정, 구독에 영향을 주는가
- 무엇을 확인하고 어떤 증거를 남겨야 하는가

## 2. 절대 원칙

이 블로그의 모든 글은 아래 원칙을 우선한다.

- 뉴스 제목을 문장만 바꿔 다시 쓰지 않는다.
- 기사 여러 개를 이어 붙인 요약문으로 끝내지 않는다.
- 독자에게 행동 기준, 체크리스트, 예시, 계산, 비교 중 최소 2개 이상을 준다.
- 제목, 본문, FAQ, 해시태그가 같은 주제를 말해야 한다.
- 지원금, 세금환급, 환불, AI, 플랫폼, 생활비 표현을 한 글 안에서 섞지 않는다.
- 불확실한 내용은 단정하지 않고 "공식 공지 기준", "확인 필요", "아직 확인되지 않았다"로 분리한다.
- 정책, 의료, 법률, 금융 고위험 주제는 공식 확인 경로를 반드시 넣는다.
- 루머, 사생활, 혐오, 선정성, 자극적 사건사고는 조회수가 가능해도 발행하지 않는다.
- 자동화 글일수록 사람 편집자가 판단한 듯한 구체 문장과 실제 독자 상황을 넣는다.

## 3. 운영 스킬 맵

### 3-1. 에디터 스킬

역할: 뉴스나 이슈를 독자 문제로 다시 번역한다.

핵심 질문:

- 이 이슈는 독자에게 어떤 손해, 이득, 불안, 선택을 만든다?
- 독자가 지금 검색할 질문은 무엇인가?
- 이 글을 읽고 오늘 바로 할 수 있는 일은 무엇인가?
- 제목이 약속한 내용을 본문이 실제로 지키는가?

금지:

- "관심이 쏠리고 있습니다"로 끝나는 글
- "공식 확인하세요"만 반복하는 글
- 제목은 구체적인데 본문은 일반론인 글

### 3-2. 주제 발굴 스킬

역할: 오늘 다룰 가치가 있는 이슈를 고른다.

점수 기준:

| 항목 | 기준 |
| --- | --- |
| 검색 의도 | 사람들이 실제로 검색할 질문인가 |
| 돈/손해 영향 | 비용, 환급, 지원, 위약금, 손해와 연결되는가 |
| 대중성 | 특정 소수가 아니라 넓은 독자에게 해당되는가 |
| 새로움 | 오늘 다룰 이유가 있는가 |
| 실전 가치 | 예시, 계산, 비교, 체크리스트로 바꿀 수 있는가 |
| 브랜드 적합성 | 생활 의사결정 블로그 정체성과 맞는가 |
| AI 인용 가능성 | Answer-first, FAQ, 조건 정리로 인용될 만한가 |
| 콘텐츠 수명 | 하루가 지나도 검색 가치가 남는가 |
| 내부 연결 가능성 | 관련 글, 후속 글, 비교 글로 확장 가능한가 |
| 리스크 감점 | 정책, 선정성, 루머, 의료/법률 단정, 클릭베이트 위험 |

주제 선택에서 가장 중요한 기준:

- 단순 화제성보다 "독자가 무엇을 확인해야 하는가"가 먼저다.
- 검색량이 커도 실익이 약하면 제외한다.
- 리스크가 크면 총점이 높아도 발행하지 않는다.
- 같은 topic_group이 반복되면 쿨다운을 둔다.

### 3-3. 제목 설계 스킬

좋은 제목 공식:

> 독자의 문제 + 손해/궁금증 + 얻는 이득

좋은 제목의 조건:

- 주제명이 구체적이다.
- 독자의 손해나 궁금증이 보인다.
- 읽고 얻는 이득이 보인다.
- 언론사 제목처럼 보이지 않는다.
- 본문이 제목의 약속을 실제로 지킨다.

좋은 예:

- 환불 지연 때 소비자가 먼저 남겨야 할 증거
- 구형폰 카카오톡 지원 종료, 내 폰도 해당될까
- 세금 환급금 조회 전 홈택스에서 먼저 볼 3가지
- 넷플릭스 신작 반응이 갈린 이유, 시청자가 먼저 본 3가지
- 한성숙 전 네이버 대표, 250억 재산 신고...진짜 봐야 할 지점은?

강한 감점 또는 금지 제목:

- 충격
- 경악
- 발칵
- 소름
- 역대급
- 난리났다
- 무조건
- 절대
- 사람들이 놓친 이면
- 진짜 변수는 따로 있다
- 유행 뒤에 숨은 돈의 흐름
- 결국 누가 더 내나
- 사생활 논란 총정리
- 루머 진짜일까

실제 운영에서 추가로 막아야 할 제목 오류:

- 언론사 시리즈명 누수: "단독", "속보", "뉴스분석", "오늘의 이슈" 같은 접두어가 제목에 남는 경우
- 깨진 조사: 받침 있는 단어 뒤에 "가"가 붙는 등 한국어 조사가 어색한 경우
- 외국 문자 혼입: 키릴, 그리스, 아랍, 일본 가나 등 한국어 제목에 뜬금없는 문자가 섞이는 경우
- 출처 제목 조각 누수: RSS 원문 제목의 괄호, 매체명, 시리즈명이 남는 경우
- 바이럴 이슈에 통신요금제 같은 생활정보 주제가 잘못 라우팅되는 경우

### 3-4. 본문 설계 스킬

기본 구조:

1. 첫 150자 안에 핵심 답변
2. 3줄 요약
3. 독자 상황 예시
4. 실제 계산, 비교, 체크리스트
5. 사람들이 놓치는 함정
6. FAQ 3개 이상
7. 한 줄 결론
8. 관련 내부 링크
9. 이미지 프롬프트 또는 대표 이미지 설명

본문 원칙:

- 첫 문단에서 바로 답한다.
- 뉴스 배경 설명은 짧게 제한한다.
- 독자가 오늘 확인할 행동을 명확히 쓴다.
- 공식 페이지, 앱 설정, 고객센터, 증빙자료 등 확인 경로를 제시한다.
- 불확실한 내용은 확인 필요로 분리한다.
- 모바일 기준으로 문단을 짧게 쓴다.
- 표, 체크리스트, FAQ는 장식이 아니라 판단 도구로 쓴다.

### 3-5. AEO/GEO 스킬

목표: Google AI Overviews, 검색 스니펫, AI 답변 엔진이 이 글을 실용적인 답변 후보로 인식하게 한다.

필수 블록:

- AI Overview용 짧은 정답 블록
- 독자 의도별 Q&A
- 확인된 것과 아직 확인할 것 분리
- 출처 신뢰 블록
- FAQPage JSON-LD
- BlogPosting JSON-LD

작성 원칙:

- 답은 앞에 둔다.
- 숫자, 조건, 날짜, 대상, 예외를 명확히 쓴다.
- 정책/지원금 글은 금액, 대상, 기간, 신청 경로, 서류, 사후 확인을 분리한다.
- FAQ 질문은 "지원금 신청 방법" 같은 generic 문장이 아니라 실제 주제명을 포함한다.

### 3-6. 발행 안전 스킬

역할: 실제 운영 데이터와 발행 히스토리를 망치지 않게 관리한다.

원칙:

- `data/publish_history.json`은 단순 로그가 아니라 편집 기억이다.
- dry-run, 삭제, 404, dereferenced, live verification 실패 기록은 실제 성공 발행으로 계산하지 않는다.
- live-verified 기록은 이후 감사 기준이 강화되어 일부 경고가 생겨도 실제 발행으로 본다.
- 히스토리 보정은 반드시 dry-run과 백업을 먼저 거친다.

운영 명령 예:

```powershell
python tools/sync_publish_history.py --dry-run --verify-live-urls
python tools/sync_publish_history.py --verify-live-urls
```

스케줄 운영 기준:

- GitHub Actions는 `17,47 * * * *`에 깨운다.
- 실제 발행 슬롯은 KST `00:13`, `08:13`, `16:13`으로 제한한다.
- `tools/should_run_news_schedule.py`가 고정 슬롯 게이트 역할을 한다.
- 발행 모드는 `NEWS_PUBLISH_MODE=publish`, `AUTO_PUBLISH=true`일 때만 실제 발행한다.
- 로컬 테스트는 기본적으로 `DRY_RUN=true`, `NEWS_PUBLISH_MODE=dry_run`을 쓴다.

## 4. 콘텐츠 축과 글 유형

### 4-1. 5개 카테고리 축

| 돈 되는 이슈 | 지원금, 환급, 생활비, 수수료에서 손해를 줄이는 기준 |
| 생활 선택 기준 | 소비자 피해, 유행 소비, 일상 선택에서 바로 쓸 판단 기준 |
| 디지털 생존법 | 앱, 계정, 결제, 백업, 서비스 종료에 대응하는 기준 |

## 5. 유형별 골든 구조

### 5-1. 지원금형

필수로 답할 것:

- 누가 대상인가
- 언제까지 신청해야 하는가
- 어디서 신청하거나 확인하는가
- 얼마를 받을 수 있는가
- 어떤 서류나 인증이 필요한가
- 중복 지원 제한이 있는가
- 신청 완료 후 무엇을 보관해야 하는가

금지:

- 지원금 글에 세금환급, 바우처, 가맹점 표현을 섞는 것
- 공식 공고 없이 금액과 대상을 단정하는 것

### 5-2. 세금환급형

필수로 답할 것:

- 환급 대상
- 조회 경로
- 홈택스/손택스 여부
- 신고 또는 신청 경로
- 환급 계좌
- 지급 예상 시점
- 지연 원인
- 공식 확인처

금지:

- 사용처
- 바우처
- 가맹점
- 지역상품권
- 지급 방식 단정

### 5-3. 환불/소비자 피해형

필수로 답할 것:

- 먼저 남길 증거
- 판매자/플랫폼/카드사 대응 순서
- 접수 번호와 날짜 기록 방법
- 결제 취소 확인 경로
- 답변 지연 시 다시 문의하는 방법

좋은 제목 예:

- 환불 지연 때 소비자가 먼저 남겨야 할 증거
- 결제 취소가 늦어질 때 카드사에 먼저 확인할 것

### 5-4. 플랫폼 변경형

필수로 답할 것:

- 내 기기, OS, 계정, 결제, 백업에 어떤 영향이 있는가
- 변경 전후 차이
- 지원 종료일 또는 적용일
- 지금 확인할 설정
- 공식 공지나 고객센터 확인 경로

### 5-5. AI 업무형

필수로 답할 것:

- 기능 소개보다 업무 흐름에서 무엇이 바뀌는가
- 누가 쓰면 좋은가
- 기존 업무와 바뀐 업무의 차이
- 바로 써볼 프롬프트 또는 설정
- 회사 기밀 입력 금지, 자동 요약 검수, 출처 재확인 같은 주의점

### 5-6. viral_issue_decode형

필수로 답할 것:

- 지금 어떤 작품, 경기, 인물, 커뮤니티 이슈가 화제인가
- 사람들이 왜 클릭하는가
- 반응이 갈린 포인트 3개는 무엇인가
- 이 이슈가 팬덤 소비, OTT 전략, 플랫폼 알고리즘, 티켓팅 구조 중 어디와 연결되는가
- 다음에 확인할 포인트는 무엇인가
- evergreen 내부링크로 어디에 연결할 수 있는가

금지:

- 확인 안 된 열애, 이혼, 불륜, 사생활 루머
- 미성년자 중심 자극 이슈
- 외모 비하, 악플 유도
- 범죄 피해자 신상 추측
- "충격 근황", "결국 터졌다", "소름 돋는 이유" 제목

## 7. HTML과 디자인 운영 규칙

Blogspot 글은 독자 화면에서 깨지지 않는 것이 최우선이다.

### 7-1. 본문 HTML

- 가능한 한 `<article class="yomi-clean-post">` 안에서 본문을 구성한다.
- 문서 전체 껍데기인 `<!doctype>`, `<html>`, `<head>`, `<body>`, `<title>`, `<meta>`가 본문에 그대로 들어가면 차단한다.
- 제목은 Blogger 테마가 렌더링하므로 본문 안의 중복 H1은 신중하게 다룬다.
- FAQ와 JSON-LD 내용은 visible한 본문 내용과 맞아야 한다.

### 7-2. 내부 링크

내부 링크는 체류 시간과 크롤링을 돕지만, 잘못 넣으면 신뢰를 깎는다.

규칙:

- 실제 발행된 Blogspot URL만 사용한다.
- dry-run, 삭제, 404, dereferenced, unverified URL은 쓰지 않는다.
- `blog-post_123.html`처럼 약한 자동 slug는 가능한 한 피한다.
- 같은 독자가 다음에 읽을 가능성이 높은 글을 우선한다.
- 내부 링크 섹션은 한 번만 넣는다.
- "같이 보면 좋은 글" CSS 장식은 문자 깨짐을 피하기 위해 `content:"->"` 같은 특수문자 의존을 줄이고, border/rotate 같은 CSS 도형으로 처리한다.

최근 운영 메모:

- `.yomi-internal-links li:before`는 깨질 수 있는 화살표 글자 대신 CSS border + rotate 방식으로 표시한다.
- Blogger 렌더링에서 HTML entity가 이중 escape되면 독자 화면에 깨진 코드가 보일 수 있으므로 감사에서 차단한다.

### 7-3. 이미지

- 대표 이미지는 제목과 내용이 일치해야 한다.
- 1200px 이상 이미지를 권장한다.
- 클릭베이트 이미지는 금지한다.
- 돈, 앱, 신청서, 설정 화면, 영수증, 체크리스트처럼 독자 상황을 시각화한다.
- 특정 인물이나 실제 얼굴을 암시해야 하는 고위험 이미지는 피한다.

## 8. 품질 게이트 체크리스트

발행 전 반드시 PASS해야 하는 기준:

- `publish_quality_gate.passed == true`
- `publish_ready == true`
- `geo_ready == true`
- `sge_ready == true`
- fallback/test 후보가 publish 모드로 올라오지 않아야 한다.
- 제목에 구체 명사 또는 원문 핵심 entity가 있어야 한다.
- 제목과 본문 핵심어가 맞아야 한다.
- 오늘성, 이슈 구체성, 원문 이슈 보존, 독자 질문 가능성이 기준 이상이어야 한다.
- FAQ, FAQPage JSON-LD, BlogPosting JSON-LD가 있어야 한다.
- source trust block이 있어야 한다.
- reader value score와 article focus score가 낮으면 차단한다.
- labels와 hashtags에 언론사명, URL, 금지 조각이 들어가면 차단한다.
- body 안에 "도입", "해시태그" 같은 내부 섹션 라벨이 사용자에게 노출되면 차단한다.
- broken HTML entity, double escape, 세미콜론 없는 entity가 보이면 차단한다.

차단 후보 예:

- fallback/test 후보가 실제 publish 모드로 올라온 경우
- 정책 지원금 글인데 공식 출처, 대상, 금액, 기간, 신청 경로, 서류가 부족한 경우
- 세금환급형에 사용처, 바우처, 가맹점 표현이 섞인 경우
- 환불이 아닌 개인정보/보안 이슈에 환불 제목이 붙은 경우
- viral_issue_decode인데 루머나 사생활 중심인 경우
- 제목은 구체적인데 본문이 "공식 확인하세요" 반복으로 끝나는 경우
- FAQ가 generic해서 실제 주제명을 포함하지 않는 경우

## 9. 운영 데이터와 발행 기록 규칙

`data/publish_history.json`은 발행 기록 이상의 의미가 있다.

역할:

- 최근 발행 주제 중복 방지
- topic_group rotation
- 내부 링크 후보 생성
- 품질 경고 학습
- post-publish audit 이슈 추적
- 스케줄 슬롯 성공 여부 판단

절대 원칙:

- 실제 운영 데이터를 고칠 때는 먼저 dry-run 리포트를 본다.
- 쓰기 전 백업을 만든다.
- 삭제/404/dereferenced 기록을 되살리지 않는다.
- live URL 검증 없이 dry-run 기록을 실제 발행으로 바꾸지 않는다.
- `.env`, 토큰, OAuth, GitHub secrets는 커밋하지 않는다.

권장 점검:

```powershell
git status -sb
python scripts/validate_news_env.py
python tools/sync_publish_history.py --dry-run --verify-live-urls
```

라이브 발행 전:

```powershell
python scripts/validate_news_env.py --strict
```

## 10. 다른 주제 블로그에 적용하는 법

요미 블로그의 핵심은 주제가 아니라 구조다.  
다른 블로그로 옮길 때는 아래를 새로 정의한다.

### 10-1. 반드시 새로 정의할 것

- 주요 독자
- 독자가 잃기 쉬운 돈, 시간, 기회, 신뢰, 건강, 생산성
- 글을 읽고 바로 해야 할 행동
- 고위험 주제와 금지 표현
- 공식 출처 스택
- 좋은 제목 공식
- 내부 링크 구조
- 발행 기록 관리 방식

### 10-2. 예시: 개인 금융 블로그

정체성:

> 복잡한 금융 뉴스를 직장인의 월급, 대출, 세금, 소비 선택 기준으로 번역하는 블로그

글 유형:

- 금리 변경 해석형
- 대출 조건 비교형
- 세금/공제 체크형
- 카드/구독비 절약형
- 금융 사기 실수 방지형

금지:

- 투자 수익 보장
- 특정 종목 매수 권유
- 검증 안 된 수익률 단정

### 10-3. 예시: 지역 생활 혜택 블로그

정체성:

> 지자체 공고와 생활 혜택을 주민이 바로 신청할 수 있는 체크리스트로 바꾸는 블로그

필수:

- 지역명
- 대상
- 신청 기간
- 신청 경로
- 필요 서류
- 문의처
- 중복 지원 여부

금지:

- 다른 지역 공고 혼합
- 종료된 지원사업 재활용
- 공식 공고 없는 금액 단정

### 10-4. 예시: 엔터테인먼트/OTT 블로그

정체성:

> 작품과 스타 이슈를 루머가 아니라 시청자 반응, 플랫폼 전략, 팬덤 소비 구조로 해석하는 블로그

필수:

- 이슈 한 줄 요약
- 반응이 갈린 포인트 3개
- 플랫폼/팬덤/소비 구조 해석
- 다음 확인 포인트
- 안전한 FAQ

금지:

- 사생활 추측
- 미성년자 자극 이슈
- 외모 비하
- 악플 유도

### 10-5. 예시: 제품/서비스 비교 블로그

정체성:

> 제품 홍보가 아니라 구매자가 돈과 시간을 덜 낭비하도록 선택 기준을 주는 비교 블로그

필수:

- 비교 기준
- 가격/요금
- 숨은 비용
- 환불/AS/해지 조건
- 어떤 사람에게 맞는지
- 사지 않아도 되는 경우

금지:

- 광고 문구 복붙
- 협찬 표기 누락
- 단점 없는 추천

## 11. 새 AI에게 넘길 최소 브리핑

시간이 없을 때는 아래만 붙여넣어도 된다.

```text
이 블로그는 뉴스 요약 블로그가 아니라, 오늘 이슈를 독자의 돈, 시간, 불안, 선택 기준으로 번역하는 한국어 생활 의사결정 블로그다.

주요 독자는 30~50대 직장인과 실용적인 소비자다.

모든 글은 첫 150자 안에 답을 줘야 하고, 독자가 바로 할 행동을 최소 3개 제시해야 한다. 제목, 본문, FAQ, 해시태그는 같은 주제를 말해야 한다. 행동 기준, 체크리스트, 예시, 계산, 비교 중 최소 2개 이상을 포함해야 한다.

좋은 제목은 "독자의 문제 + 손해/궁금증 + 얻는 이득"이다. 충격, 소름, 난리났다, 역대급, 무조건, 절대, 사람들이 놓친, 진짜 변수 같은 표현은 금지한다.

지원금, 세금환급, 환불, AI, 플랫폼, 생활비 표현을 한 글 안에서 섞지 않는다. 정책/지원금/환급 글은 공식 확인처, 대상, 기간, 금액, 신청 경로, 서류, 사후 확인을 분리한다.

viral_issue_decode는 연예/스포츠/OTT 이슈를 루머가 아니라 반응 구조와 플랫폼/팬덤/소비 흐름으로 해석한다. 사생활, 악플, 미성년자 자극, 확인 안 된 루머는 금지한다.

발행 전에는 제목-본문 일치, FAQ/JSON-LD, source trust block, 내부 링크 실존 여부, HTML 깨짐, 해시태그 일치, live URL 히스토리 상태를 확인해야 한다.
```

## 12. 발행 전 1페이지 체크리스트

주제:

- 오늘 다룰 이유가 있는가
- 독자의 돈, 시간, 불안, 선택과 연결되는가
- 공식 확인 경로가 있는가
- 최근 발행 주제와 과하게 겹치지 않는가
- AdSense/Discover 리스크가 낮은가

제목:

- 구체 명사가 있는가
- 독자의 문제나 궁금증이 보이는가
- 본문이 제목 약속을 지키는가
- 금지어가 없는가
- 외국 문자, 출처 시리즈명, 깨진 조사가 없는가

본문:

- 첫 150자 안에 답이 있는가
- 바로 할 행동 3개가 있는가
- 예시, 비교, 계산, 체크리스트 중 2개 이상이 있는가
- 확인된 것과 확인할 것이 분리됐는가
- FAQ가 실제 주제명을 포함하는가
- 해시태그가 content_type과 맞는가

기술:

- FAQPage JSON-LD가 있는가
- BlogPosting JSON-LD가 있는가
- meta description이 본문에만 있지 않은가
- 내부 링크가 살아 있는 URL인가
- HTML entity나 CSS content 문자가 깨지지 않는가
- dry-run/test/fallback 후보가 publish로 올라오지 않았는가

운영:

- `data/publish_history.json`이 실제 발행 상태와 맞는가
- 삭제/404/dry-run 기록이 성공 발행으로 계산되지 않는가
- 백업 없이 운영 데이터를 고치지 않았는가
- `.env`와 secrets가 커밋 대상에 없는가

## 13. 이 파일을 업데이트하는 기준

아래 일이 생기면 이 파일도 함께 갱신한다.

- 좋은 글 패턴이 새로 발견된 경우
- 실제 발행에서 깨진 제목, 깨진 HTML, 깨진 내부 링크가 발견된 경우
- 품질 게이트에 새 차단 조건이 추가된 경우
- Google Search, Discover, AdSense 기준에 맞춰 운영 방식을 바꾼 경우
- 다른 주제 블로그로 이식하면서 재사용 가능한 프롬프트가 늘어난 경우

이 문서는 완성본이 아니라 운영 자산이다.  
블로그가 성장할수록, 좋은 글과 나쁜 글의 차이가 더 선명해질수록, 이 파일도 더 날카로워져야 한다.
