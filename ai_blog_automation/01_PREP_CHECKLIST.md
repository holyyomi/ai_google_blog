## 2. 독자 정의

아래 중 가장 중요한 독자 1개를 먼저 고른다.

- 직장인: 업무 요약, 보고서, 엑셀, 회의록, 이메일 자동화에 관심
- 블로거/콘텐츠 운영자: 글쓰기, 썸네일, SEO, 콘텐츠 재활용, 수익화에 관심
- 1인 사업자/마케터: 광고 문안, 고객 응대, 랜딩페이지, CRM 자동화에 관심
- 개발자/자동화 실무자: API, 에이전트, 워크플로, n8n, Make, Zapier에 관심
- 초보 사용자: ChatGPT, Gemini, Claude를 어떻게 시작할지 궁금

권장 시작점:

> 30~50대 직장인과 1인 콘텐츠 운영자를 핵심 독자로 잡고, AI 기능을 "업무 시간 절약, 콘텐츠 생산, 수익화, 보안 리스크" 기준으로 번역한다.

## 3. 새 블로그의 핵심 약속

AI 블로그가 독자에게 약속해야 할 것:

- 새 AI 기능이 내 업무에 어떤 영향을 주는지 알려준다.
- 어떤 AI 도구를 언제 써야 하는지 선택 기준을 준다.
- 바로 복사해 쓸 수 있는 프롬프트와 워크플로를 제공한다.
- 비용, 무료/유료 제한, 한국어 지원, 보안 리스크를 확인하게 한다.
- AI가 만든 결과를 그대로 믿지 않고 검수하는 방법을 알려준다.

금지할 방향:

- AI 뉴스 제목만 바꿔 쓰는 글
- "대단하다", "혁신이다" 감상문
- 실제로 해보지 않은 수익 인증
- 검증 안 된 가격, 성능, 출시일 단정
- 도구 링크를 가짜로 만드는 것
- 특정 AI 도구를 무조건 추천하는 것

## 4. 운영자가 준비할 계정과 키

필수:

- 새 Blogspot 블로그
- 새 Blogger blog ID
- Blogger OAuth client ID
- Blogger OAuth client secret
- Blogger refresh token
- Google AI API key
- OpenAI API key
- 새 GitHub repository

선택:

- Naver Search API key
- Google Custom Search API key/CX
- Tavily/Exa/Firecrawl API key
- 대표 이미지 URL 또는 이미지 생성 API
- Search Console
- AdSense

## 5. 절대 복사하면 안 되는 기존 데이터

새 블로그 repo로 가져가면 안 되는 것:

- `.env`
- 기존 `data/publish_history.json`
- 기존 `state/news_published_history.json`
- `runs/news_*`
- `logs/`
- 기존 블로그의 내부 링크 캐시
- 기존 Blogspot ID와 OAuth token
- 기존 블로그 URL 기준 sitemap/audit 결과

새 블로그에서 새로 만들어야 할 것:

- 빈 발행 히스토리
- 새 내부 링크 기준
- 새 라벨/해시태그 체계
- 새 주제 쿨다운 기록
- 새 post-publish audit 기록

## 6. 운영 전에 정해야 할 정책

아래 질문에 답해두면 Codex가 코드 개편을 안전하게 진행할 수 있다.

1. 새 블로그는 AI 뉴스 중심인가, AI 활용법 중심인가?
2. 자동 발행은 처음부터 할 것인가, 수동 검수 후 할 것인가?
3. 다룰 AI 도구 범위는 어디까지인가?
4. 개발자용 API 글도 다룰 것인가?
5. 수익화 글을 다룰 때 실제 수익 인증이 없으면 어떻게 표현할 것인가?
6. 특정 제품 추천 글에서 협찬/광고 가능성이 생기면 어떻게 표기할 것인가?
7. 회사 업무 자동화 글에서 보안/기밀 입력 금지 문구를 어디에 넣을 것인가?
